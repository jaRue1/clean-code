# Successive Refinement

> "It is not enough for code to work. Code that works is often badly written and disorganized."

## Takeaways

<pre>
Programming is a craft more than it is a science. To write clean code, you must first write dirty code and then clean it. This is not an excuse for sloppy work; it is an acknowledgment that refinement is part of the process.
</pre>

<pre>
Code is never right the first time. The first draft is just that: a draft. Getting it to work is the first step. Cleaning it up is the necessary second step that too many developers skip.
</pre>

<pre>
Incremental improvement is key. Do not try to design the perfect solution up front. Start with something that works, then refine it step by step, improving the structure with each pass.
</pre>

<pre>
First make it work, then make it clean. The mistake is to stop at "it works." Working code that is messy will deteriorate over time, becoming harder to modify and more prone to bugs with every change.
</pre>

<pre>
Refactoring is an ongoing discipline, not a one-time event. Every time you touch code, leave it a little better than you found it. Small, continuous improvements prevent rot and keep the codebase healthy.
</pre>

<pre>
Small steps with tests. When refactoring, make one small change at a time and run the tests after each change. This way you always know what broke and you can easily revert if needed.
</pre>

## Opinions

<pre>
It is not enough for code to work. Code that works is often badly written and disorganized. Professionals know that this is the starting point, not the finish line.
</pre>

<pre>
The act of writing clean code is more like successive refinement than brilliant first-attempt design. The key is having the discipline to go back and clean up.
</pre>

<pre>
If we all checked in our code a little cleaner than when we checked it out, the code simply could not rot. The cleanup does not have to be something big. Change one variable name for the better, break up one function that is a little too large, eliminate one small bit of duplication.
</pre>
