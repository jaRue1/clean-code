# JUnit Internals

> "No module is immune from improvement, and each of us has the responsibility to leave the code a little better than we found it."

## Takeaways

<pre>
Even well-known, widely used frameworks like JUnit have room for improvement.
</pre>

<pre>
Apply the Boy Scout Rule to any code you read: leave it cleaner than you found it.
</pre>

<pre>
Refactoring steps: improve names, extract methods, reduce complexity, clean conditionals.
</pre>

<pre>
Small improvements compound over time into significant quality gains.
</pre>

<pre>
Code should be continuously polished, not written once and abandoned.
</pre>

<pre>
The ComparisonCompactor class in JUnit works correctly but can be made more expressive through renaming and restructuring.
</pre>

<pre>
Negatives in conditionals are harder to understand than positives. Prefer positive checks.
</pre>

<pre>
Encapsulating conditionals behind well-named methods makes code read like prose.
</pre>

</pre>
