# Clean Code

> Clean code reads like well-written prose.

## Takeaways

<pre>
The only way to make the deadline; the only way to go fast, is to keep the code as clean as possible at all times.
</pre>

<pre>
LeBlanc's Law: Later equals never.

We have all said we would go back and clean it up later. We never did.
</pre>

<pre>
Clean code always looks like it was written by someone who cares.
</pre>

<pre>
You know you are working on clean code when each routine turns out to be pretty much what you expected.
</pre>

<pre>
The ratio of time spent reading vs writing code is well over 10:1.

We are constantly reading old code as part of the effort to write new code.
</pre>

<pre>
The Boy Scout Rule: Leave the campground cleaner than you found it.

If we all checked in our code a little cleaner than when we checked it out, the code simply could not rot.
</pre>

<pre>
Clean code does one thing well. Bad code tries to do too much.
</pre>

<pre>
Clean code is focused. Each function, each class, each module exposes a single-minded attitude that remains entirely undistracted by the surrounding details.
</pre>

## Opinions

<pre>
We are authors. And one thing about authors is that they have readers.
Indeed, authors are responsible for communicating well with their readers.
</pre>

<pre>
The only way to go fast is to keep the code clean.
Every time you take a shortcut, you slow down everyone who reads the code after you.
</pre>

<pre>
It is not the language that makes programs appear simple. It is the programmer that make the language appear simple.
</pre>
