# Unit Tests

> "Test code is just as important as production code."

## Takeaways

<pre>
The Three Laws of TDD:
1. You may not write production code until you have written a failing unit test.
2. You may not write more of a unit test than is sufficient to fail, and not compiling is failing.
3. You may not write more production code than is sufficient to pass the currently failing test.
</pre>

<pre>
Test code is as important as production code. It is not a second-class citizen. It requires thought, design, and care.
</pre>

<pre>
What makes a clean test? Readability. Readability. Readability. Clarity, simplicity, and density of expression.
</pre>

<pre>
One assert per test, or more precisely, one concept per test. Each test function should test a single concept.
</pre>

<pre>
F.I.R.S.T. principles for clean tests:
Fast: Tests should run quickly.
Independent: Tests should not depend on each other.
Repeatable: Tests should be repeatable in any environment.
Self-Validating: Tests should have a boolean output; they either pass or fail.
Timely: Tests should be written just before the production code that makes them pass.
</pre>

## Opinions

<pre>
Test code is just as important as production code. It requires thought, design, and care. It is not a second-class citizen.
</pre>

<pre>
If you let the tests rot, then your code will rot too. Keep your tests clean.
</pre>
